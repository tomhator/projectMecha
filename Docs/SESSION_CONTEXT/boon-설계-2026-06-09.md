# boon 설계 — 방향 합의 & 선행 의존성 진단 (2026-06-09)

## 요약
> 어제(06-08) 던전 맵 재설계에서 "내일"로 미뤄둔 **교리 노드 런-한정 boon**의 방향을 대화로 잡았다. boon은 하데스 풀카피가 아니라 **"미리 세팅해온 코어 어빌리티를 강화"** 하는 쪽으로 정리됐다. 다만 강화의 훅이 될 **교리(정비공/해체공)가 코드에 전혀 없고**, 코어 어빌리티 모델도 얇다는 걸 확인 → boon 설계는 **"2교리 vs 3축 미정합" 해결을 선행 의존성으로** 가지며, 오늘은 거기서 멈추고 **boon 본 설계는 다음 세션으로 이월**한다.

---

## 작업 내용

### 합의된 boon 방향 (사용자 확정)

1. **구조 — 하데스 풀카피 아님.** 맵에 최대 4개 갈림길/선택지가 보이는데, **그중 일부가 교리에 따른 패시브를 부여**한다는 아이디어. 교리를 가진 노드는 **전투일 수도, 조우일 수도, 상자일 수도** 있다.
2. **콤보 — 약한 시너지만.** 하데스처럼 boon들이 빽빽하게 연계되는 정도는 아니고, **같은 교리를 여러 번 선택하면 약한 수준의 시너지**가 생기는 정도.
3. **boon = 미리 세팅해온 코어 어빌리티 강화.** 던전 시작 전에 유저가 본인 빌드 계획에 맞게 **코어 어빌리티를 세팅**하게 유도하고, 런 중 교리 노드가 그 세팅을 **강화**한다. 하데스의 "뽐(Pom)"이 기존 boon을 강화하듯, 여기선 노드가 **이미 장착한 코어 동사**를 강화. **새 인터랙트는 추가하지 않는 방향**(선택은 ⓐ던전 전 빌드 세팅 + ⓑ맵 경로에서 이미 끝남).
4. **강화 대상 — 노드 종류(전투/조우/상자)별로 다름.** 어떻게 다른지는 **아직 미정** (코어 어빌리티 현황부터 확인하기로 함 → 아래).

> 이 모델의 장점: 별도 보상 시스템을 안 늘리고 기존 맵 설계(경로=보급 전략)에 boon을 얹음 → **"시스템 심화 금지"** 가드레일 정합. "같은 교리 반복=시너지, 섞으면 밋밋" → "이번 런 한 교리 몰빵" 긴장 자연 발생.

### 코어 어빌리티 현황 진단 (코드 + 문서 조사)

**출격 빌드 = 3칸** (`Docs/AbilityTreeSystem.md` "출격 선택 영역"):

| 빌드 슬롯 | 선택 | 실체 |
|----------|------|------|
| ① 기본 공격 택1 | 화력/민첩/방어 | 입자 광선·속사 사격·간이 방벽 (`Resources/Skills/skill_core_*.tres`) — 상시·무손상 |
| ② 어빌리티 트리 티어별 1개 | 거점 크레딧 영구 연구 | T3 고출력루프/방호순환/전력분배 · T4 비상점화/격실차단/잔해재배치 · **T5 캡스톤** 소모성화력/생존코어/현장개조 (`Resources/AbilityTree/*.tres`) |
| ③ 파츠 활용 어빌리티 택1 | 조건부 특수행동 | 긴급 교체·파손 투척·임시 덧대기·현장 수리·드론 정비 |

**5개 코어 동사 → 교리 매핑** (전투 spec §3.2 "코어 어빌리티 ★" 컬럼):
- **정비공("막는다")** = 수리 · 덧대 (+ 기본공격 간이 방벽)
- **해체공("태운다")** = 투척 · 교체 · 소모 (+ 기본공격 입자/속사)

**코드 실태** (`Resources/SkillData.gd`):
```gdscript
enum CoreSkillRole  { NONE, BASIC_ATTACK, PART_ABILITY }      # 코어 분류 3종뿐
enum PartAbilityKind { NONE, EMERGENCY_SWAP, BROKEN_THROW, SCRAP_PATCH }  # 교체·투척·덧대
```

### 발견된 갭 (boon 설계 전 정리 필요)

- **갭① 2교리 vs 3축 미정합 (★ 최우선 토대).** boon·교리는 **정비공/해체공 2교리** 전제인데, 어빌리티 트리는 **공격/방어/유틸 3축**(T5 캡스톤 "소모성화력/생존코어/현장개조 교리")으로 돼 있어 1:1로 안 맞는다.
- **갭② 교리가 코드에 0건.** grep 결과 doctrine/교리/정비공/해체공 **코드 전무** → 기획 문서에만 존재, 미구현. boon이 걸릴 **교리 태그·동사 카테고리 훅이 코드에 없음.**
- **갭③ 코어 어빌리티 분류가 얇음.** `core_skill_role` 플래그 3종(NONE/기본공격/파츠활용)뿐. 교리 묶음·전용 프레임 없음. **"수리"가 `PartAbilityKind`에서 누락**(정비공 핵심 동사인데).
- **갭④ "던전 전 빌드 세팅" 화면 부재.** CoreSelect는 트리 노드 선택만, 전투 UI(`Scenes/UI/CombatUI.gd`)는 `get_available_skills()`를 AP 필터해 **한 컨테이너에 평면 나열**. boon이 "미리 세팅한 빌드 강화"라면 이 세팅 화면이 전제 조건.

## ▶ 다음 세션 재개 지침 (boon 설계 착수 시 이 순서로)

> **0단계 — 이 문서 + 맵 스펙 §8.3/§9 다시 읽기.** [[superpowers/specs/2026-06-08-맵화면-재설계-design]] §8.3(교리 노드+런 boon), §9(미결). 합의된 방향 4가지(위 "합의된 boon 방향") 유지.

1. **갭① 먼저 결정 (토대).** "2교리(정비공/해체공) vs 3축(공격/방어/유틸)"을 어떻게 정합시킬지. 검토했던 3안:
   - **A) 3축 유지 + 교리는 태그만** — 트리는 그대로, `SkillData`에 `doctrine` 태그(중립/정비공/해체공) 추가. boon은 "정비공 동사 전부 +X"로 훅. *리스크 최소·시스템 심화 금지 부합. 단 트리(3축)·boon(2교리) 이중 언어 잔존.* ← **유력 추천(최소 변경)**
   - **B) 2교리로 트리 통합** — 공격·유틸→해체공, 방어→정비공 재편. 정합성 최고지만 트리 대수술 + 저장 포맷·CoreSelect 영향.
   - **C) 직교 2레이어** — 트리(공/방/유)=영구 "코어 성능 스탯", 교리(정비공/해체공)=런한정 "동사 운용 스타일". boon은 교리 레이어에만 작용. 하데스 "거울(영구)+신boon(런)" 구조와 정확히 일치하나 개념 한 승 늘어남.
   - *판단 보류 상태 — 사용자와 다시 논의해서 확정할 것.*

2. **갭② 해소** — 결정한 교리 정합을 `SkillData`(또는 별도 데이터)에 **교리 태그/카테고리로 명문화**. boon이 걸릴 훅 마련. "수리"를 코어 동사 분류에 포함(갭③).

3. **노드 종류 → 강화 대상 매핑 확정** (미결). 검토했던 후보:
   - **빌드 슬롯 매핑** — 전투→③파츠활용, 조우→②트리, 상자→①기본공격.
   - **교리 고정(종류=강도만)** — 노드는 교리만 보고 내 장착 교리 동사 강화, 종류는 강도·연출·대가만 차등(전투=큰 강화+위험 / 조우=중간+선택 / 상자=작은+공짜).
   - **동사 카테고리 매핑** — 전투=공격형(투척/입자), 조우=운영형(수리/분해/제작=작업대 동사), 상자=내구·방어형(덧대/쉴드).

4. **"같은 교리 반복 → 약한 시너지" 기계 확정** — 앞서 든 후보: 세트 보너스(임계치 N개) / 누적 스케일링 / (사용자 방향)=세팅 빌드 동사 누적 강화. 새 자원·새 인터랙트 금지 가드레일 유지.

5. **개별 boon 종류 설계** (위가 정해진 뒤) — 정비공(수리·덧대 증폭) / 해체공(투척·교체·소모 증폭), 희귀도 도입 여부.

6. **산출물** — boon + 교리 코어 어빌리티 통합 설계를 `Docs/superpowers/specs/YYYY-MM-DD-boon-교리코어-design.md`로. 맵 스펙 §8.3/§9에 boon 세부 채워 넣기.
